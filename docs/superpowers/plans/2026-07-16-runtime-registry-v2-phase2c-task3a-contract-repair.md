# Runtime Registry v2 Phase 2C Task 3A Contract Repair

**Status:** Approved for implementation

## 1. Problem

The Task 3 reconciler cannot safely inspect a new deterministic runtime identity because
`DriverIdentity` requires `attempt_id`, while the current repository creates that identity
inside `begin_attempt()`. Creating the attempt before inspection violates the invariant that
absence or safe adoption must be proven before a new active attempt is persisted.

Supervisor restart also lacks a purpose-specific immutable read model for the exact persisted
attempt material, the state provider cannot verify a ready existing allocation, and stop or
identity-ambiguity outcomes cannot be recorded honestly and atomically.

## 2. Goals

- Obtain a repository-owned candidate attempt ID without creating an attempt.
- Persist that exact ID only after reconciliation permits launch.
- Reload exact non-secret active-attempt provenance after Supervisor restart.
- Latch identity ambiguity atomically while preserving the active attempt as a blocker.
- Record an already-absent stop without inventing an exit code.
- Verify an existing ready state allocation without creating, hardening, quarantining, or
  deleting filesystem objects.

## 3. Non-goals

- No reconciliation decision state machine; that remains Task 3B.
- No LaunchSnapshot compiler or Compose driver; those remain Task 4.
- No bounded health, ambiguous launch recovery, or offline identity; those remain Task 6.
- No daemon, CLI, API, or Docker execution.
- No database schema migration unless a failing invariant proves one is unavoidable.

## 4. Contract

### 4.1 Attempt identity

`prepare_attempt_id(job_id)` validates the current leased start/retry job and absence of an
active attempt, then returns a closed `Identifier` generated by the repository. It performs no
database mutation. `begin_attempt(job_id, attempt_id, resolved_material)` repeats all lease,
active-attempt, and immutable-material checks and persists exactly the supplied ID before any
future driver launch.

### 4.2 Recovery material

`get_latest_attempt_material(instance_id)` returns either `None` or a frozen, closed,
purpose-specific value containing the latest attempt ID/status, exact RuntimeSpec payload digest,
and the already-persisted `ResolvedRuntimeMaterial`. Task 3B must reconcile an active latest
attempt before preparing another one and must use a terminal latest attempt as the persisted
identity gate against an out-of-band resurrection. It contains no secret value, host path, DSN,
credential, arbitrary command, mount, or network authority. Network identity remains derived by
the later closed snapshot compiler from immutable policies; it is not invented here.

### 4.3 Reconciliation block

`record_reconciliation_blocked(job_id, attempt_id, failure_code)` requires the current lease and
an exact optional active-attempt binding. In one transaction it sets the job to
`needs_reconciliation`, clears its lease, latches the instance, moves the instance lifecycle to
`failed`, and appends the audit event. It does not terminalize or mutate the active attempt, so an
unknown or mismatched present object continues to block another attempt.

### 4.4 Already-absent stop

`record_stopped(..., exit_code=None)` is valid only for an explicit stop job. It records the
attempt as stopped with a null exit code and emits `container_already_absent`. Integer exit codes
retain the existing `stopped` result. Booleans and non-integers remain invalid.

### 4.5 Existing state

`ExistingStateAllocation` is distinct from `StateAllocationReservation`. `provision()` continues
to accept only a fresh reserved allocation. `verify_existing()` accepts only a fresh ready
allocation and performs a complete read-only root, allocation, layout, identity-file, ownership,
permission, containment, and TOCTOU proof. Any failure is redacted and fail-closed; it never
hardens, creates, moves, quarantines, removes, or repairs the allocation.

## 5. Verification

- RED tests first for every new contract and invalid ordering.
- SQLite transaction tests for rollback and immutable recovery material.
- Real PostgreSQL transaction/concurrency selectors for the changed repository path.
- Windows/POSIX-compatible root tests proving existing-state validation is read-only.
- Existing Task 1 Driver and Task 2 repository suites remain green.
- Independent architecture, security, and code-quality review before commit.
